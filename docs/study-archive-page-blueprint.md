# 勉強会ログ・アーカイブ一覧ページ 設計メモ

## 1. このページの目的
- 過去の勉強会ログ、資料、録画、関連リンクを迷わず探せるようにする
- 忙しい薬剤師が短時間で「今必要な情報」に到達できるようにする
- 単なる記録置き場ではなく、継続学習の導線として機能させる

## 2. 想定ユーザー
- 薬剤師

### 薬剤師向けに重視する点
- 隙間時間でも読める一覧性
- 信頼できる情報かが一目でわかること
- 薬物療法の実務に沿ったテーマ分類
- 症例や患者情報の扱いに配慮されていること

## 3. このページで達成したい行動
1. テーマから必要なアーカイブを探す
2. キーワードや年度から絞り込む
3. 録画をすぐ再生する
4. まず何を見ればよいか判断する
5. 関連テーマのアーカイブへ移動する

## 4. まず載せるべき情報

### 一覧カードに必要な情報
- タイトル
- 開催日
- テーマ
- 概要 2〜3行
- 録画時間
- 利用できる資料種別
  - 録画
  - スライド
  - 要約ノート
  - 参考文献
- 登壇者名
- 閲覧導線ボタン

### 詳細導線で持たせたい情報
- 勉強会の目的
- 学習ポイント 3点前後
- 参考ガイドラインや論文
- 関連テーマ
- 注意事項
  - 教育目的であること
  - 個別診療の代替ではないこと
  - 症例情報は匿名化済みであること

## 5. ページ構成案

### ファーストビュー
- ページタイトル
  - 例: 「勉強会アーカイブ」
- 短い説明文
  - 例: 「薬剤師向けの勉強会録画、資料、参考リンクをテーマ別に確認できます」
- 主要CTA
  - 最新の録画を見る
  - テーマから探す

### 検索・絞り込みエリア
- キーワード検索
- テーマ
  - 例: 感染症、がん、緩和ケア、在宅、ポリファーマシー
- 資料種別
  - スライドあり
  - 要約あり
  - 参考文献あり
- 年度・開催年

### 一覧エリア
- 新しい順を基本表示
- 1件ごとにカード表示
- 各カードに「45分」「スライドあり」などの視認性の高いラベルを付ける

### 補助エリア
- 人気テーマ
- 初めて見る人向けおすすめテーマ
- 直近3回の更新

## 7. デザイン方針

### 方向性
- 落ち着きと信頼感があること
- 情報が多くても圧迫感が少ないこと
- 薬剤師向けらしい整理感は保ちつつ、無機質になりすぎないこと

### 推奨トーン
- ベースカラーは白、淡いグレー、深い青緑系
- 強調色は1色に絞る
- 危険色の赤は警告用途のみに限定する

### UIの雰囲気
- カード余白は広め
- バッジで情報を圧縮
- 見出しは明快、本文は短く
- 文字サイズはやや大きめにして可読性優先

## 8. 薬剤師向けならではの配慮

### 信頼性の見せ方
- 登壇者の所属・専門領域を明記
- 参考文献やガイドラインの有無を表示
- 更新日を表示し、情報鮮度を伝える

### 安全性の見せ方
- 「教育用コンテンツ」であることを明記
- 個別症例に関する診断・治療助言と誤認されない表現にする
- 症例提示がある場合は匿名化・加工済みと明記

### 現場導線
- スマホ閲覧前提で設計する
- 録画への導線を最優先表示する
- 「短時間で把握できる要約」は補助導線として置く

## 9. 法務・運用の必須要件

### 画面に必ず必要なもの
- フッターか固定領域に `利用規約` `プライバシーポリシー` `問い合わせ先` を置く
- 一覧ページと詳細ページの両方に `教育目的であり、個別診療の代替ではない` と明示する
- 動画を埋め込む場合は、再生前に `YouTube が読み込まれ、外部サービスへ情報が送信される場合がある` と読める日本語の注意文を出す
- 著作権や個人情報に関する削除依頼窓口を用意する

### 公開前に必ず必要なもの
- 患者情報チェック
- 登壇者の公開許可確認
- 資料 PDF / スライド / 画像の権利確認
- 外部リンク先が正しいかの確認
- 誤掲載時の停止手順

### 表現上のルール
- 治療効果の断定、優良誤認、比較優良、体験談まかせの見せ方を避ける
- タイトルや要約は `教育メモ` の文体で書き、宣伝文にしない
- 一般向けに見える画面では、薬剤や医療機器の効能を大げさに書かない

## 10. コンテンツ項目のデータ設計案

各アーカイブに最低限あるとよい項目:

```yaml

  recording:
  slides:
  notes:
  references:

```

公開データに載せなくても、内部管理では次の確認情報を必須にする:

```yaml

  speaker_consent_confirmed:
  patient_info_checked:
  copyright_basis:
  publication_approved_at:
  review_memo:
```

## 11. セキュリティ設計の前提
このプロダクトは現在 `静的公開ページ + ローカル運用` が中心ですが、将来 `認証 / API / データベース / サーバー保存` を追加する場合も見据えて、以下を設計の前提にします。

### 公開データに載せない情報
- ログイン情報、メールアドレス、氏名、施設名、内部メモなどの個人情報・機密情報は `public/` 配下に置かない
- `public/data/site-content.js` やブラウザから取得できる JSON / HTML / API レスポンスには、閲覧者全員に見えてよい情報だけを含める
- サーバーから返す情報は最小限にし、画面表示に不要な個人情報や内部識別子を返さない

### 個人情報と機密情報の扱い
- 個人情報、認証情報、権限制御に関わる判定、内部メモ、APIキーなどは `クライアントサイドではなくサーバーサイド` で扱う
- クライアント側は表示と入力補助に限定し、秘匿すべき値を保持しない
- ブラウザの開発者ツールや通信ログを見られても、個人情報や機密情報が盗み取れない構造を維持する

### 機密情報の管理
- APIキー、アクセストークン、DB 接続情報、環境変数の値は絶対にハードコードしない
- 機密情報はサーバー環境変数またはシークレット管理機構で扱い、リポジトリや公開 JS に含めない
- サンプルコード、初期値、コメント内にも実キーや実トークンを残さない

### 実装時に必須とする対策
- XSS 対策: ユーザー入力や外部データをそのまま `innerHTML` に入れず、エスケープまたはサニタイズを行う
- XSS 対策: Markdown / HTML を扱う場合は許可するタグを限定し、必要に応じて CSP を設定する
- CSRF 対策: 状態変更系のリクエストには CSRF トークン、`SameSite` 属性付き Cookie、`Origin / Referer` 検証を組み合わせる
- SQL インジェクション対策: SQL は文字列連結せず、プレースホルダ付きクエリや ORM のバインド機能を使う
- 入力値検証: サーバー側で形式、長さ、許可値を必ず検証し、クライアント側のバリデーションだけに依存しない

### 将来の拡張時に守るルール
- ログイン機能を追加する場合、メールアドレスなどのアカウント情報は本人と管理者以外に見えないことを要件にする
- 管理画面をサーバー化する場合、権限ごとのアクセス制御と監査ログを持たせる
- 公開ページ用 API と管理用 API は分離し、管理用 API を公開ページから直接呼ばせない

## 12. MVPでやること
- 一覧ページを作る
- キーワード検索を付ける
- テーマ、資料種別、年度で絞り込めるようにする
- 各テーマに最低限の概要と録画リンクを載せる
- 更新しやすいデータ形式にする
- 法務ページへの導線を用意する
- 公開前チェックを運用に組み込む

## 13. 後から足すと強い機能
- おすすめ表示
- 視聴済み管理
- よく読まれている回のランキング
- シリーズ単位のまとまり表示
- 「今月の学習テーマ」導線

## 14. 運用上の注意
- 各テーマごとにメタ情報の入力ルールを揃える
- タイトル命名ルールを決める
- 参考文献の表記ルールを統一する
- 公開前に個人情報・施設情報が残っていないか確認する
- 公開前にブラウザのネットワークタブでレスポンス内容を確認し、個人情報や内部情報が返っていないことを確認する
- 将来 API やログイン機能を追加する場合は、XSS / CSRF / SQL インジェクション対策の実装確認を公開条件にする
- 資料ごとに `自作 / 許諾あり / リンクのみ / 引用` を残す
- 削除依頼が来た時の責任者と連絡先を決める
- 漏えい時の初動を 1 枚の手順書にする

## 15. 次に決めるとよいこと
1. テーマの分類をどこまで細かく分けるか
2. まず公開する件数は何件か
3. 録画に付ける補助情報をどこまで載せるか
4. 既存の勉強会データをどう整理して持つか
5. 録画ページを外部リンクにするか、サイト内詳細ページにするか

## 16. 初回提案の結論
- まずは「薬剤師が短時間で録画を探せるアーカイブ一覧」を最優先にする
- 一覧カードの情報密度を高めつつ、見た目は静かにまとめる
- 信頼性、匿名化、教育目的であることの3点は最初から入れる
- MVPは一覧、検索、絞り込み、録画リンクまでで十分
- ただし `利用規約` `プライバシーポリシー` `外部送信の説明` は初期段階から必須とする
- 将来サーバー機能を足す場合でも、個人情報は公開データに載せず、機密情報はサーバー側だけで扱う前提を崩さない
