# アーカイブ詳細ページ 設計メモ

## 1. この設計の目的
- 一覧ページは軽いまま保つ
- 各アーカイブに動画、資料、要約、目次を載せられるようにする
- 最初から入力項目を増やしすぎず、段階的に運用できる形にする

## 2. いまの結論
最初の実装は `1つの詳細テンプレート` で十分です。

```text
public/
  index.html
  archive/
    index.html
  assets/
    app.js
    archive-detail.js
    styles.css
  data/
    site-content.js
```

想定URL:

```text
/archive/?id=archive-20260224-f1f9a4cdfb
```

理由:
- 実装が軽い
- 運用ルールを作りやすい
- 一覧ページを壊さずに追加できる

## 3. 今回の決定事項
- 第一段階は `archive/index.html?id=...` で始める
- 動画は `YouTube 埋め込み` を基本にする
- 埋め込みは詳細ページでだけ行い、一覧ページでは行わない
- `iframe` は最初から出さず、`再生する` 操作後に読み込む
- 埋め込みできない動画だけは `YouTubeで開く` を fallback にする
- 動画URLは既存の `links.recording` を優先利用し、入力項目は増やしすぎない
- 要約は `現在の summary と同じくらいの短さ` を基準にする
- 動画がある回は、YouTube の字幕 / 文字起こしを元に要約案を作る前提で考える
- 手入力は最小限にして、生成できるものは可能な範囲で自動生成する
- 一覧カードの `タイトル` は詳細ページへのリンクにする
- 一覧カードの `再生する / 参考文献` などの即時アクションは残す
- 第一段階では `詳細を見る` ボタンは増やさず、カード内の導線を増やしすぎない

補足:
- `https://www.youtube.com/watch?v=...`
- `https://youtu.be/...`

どちらの形式でも埋め込み用URLへ変換できます。

## 4. 将来の拡張方針
将来的に `各回ごとのOGP` や `共有URLの見栄え` を強くしたくなったら、次の形へ移行します。

```text
/archive/atrial-fibrillation-pharmacotherapy/
```

これは `第二段階` として考えるのが自然です。最初からここまでやる必要はありません。

## 5. ページに載せる内容
詳細ページの基本ブロックは以下を想定します。

1. タイトル、開催日、テーマ、講師
2. 概要
3. 動画導線
4. 資料導線
5. 学習ポイント
6. 目次 / タイムスタンプ
7. 関連アーカイブ
8. 教育用コンテンツの注意書き

方針:
- 動画やPDFは詳細ページでだけ扱う
- 該当データが無いブロックは表示しない

## 6. 一覧カードの導線方針
一覧カードは `即再生したい人` と `内容を確認してから見たい人` の両方に対応します。

第一段階の導線:
- タイトルを押すと詳細ページへ移動
- `再生する` は今まで通り直接再生リンク
- `参考文献` などの資料リンクも今まで通り直接開く
- カード全体はリンクにしない
- `詳細を見る` ボタンは付けない

理由:
- 主要導線が `タイトル` に集約されるので分かりやすい
- ボタンを増やしすぎず、一覧の見た目を保てる
- 今すぐ動画を見たい人の操作回数を増やさない

## 7. データ設計案
既存の `archives[]` を壊さず、追加項目だけ増やします。

```js
{
  id,
  slug,
  themeId,
  title,
  summary,
  speaker,
  date,
  updatedAt,
  duration,
  featured,
  assets,
  links,
  thumbnail,
  detail: {
    overview,
    keyPoints: [],
    chapters: [
      { time: "00:00", label: "導入" }
    ],
    materials: [
      { label: "参考資料", url: "..." }
    ],
    video: {
      provider: "youtube",
      url: "..."
    }
  }
}
```

ルール:
- `detail` は optional
- `chapters` も `keyPoints` も空でよい
- `slug` は将来の静的URL化に備えて先に持たせる
- `slug` はタイトルから自動生成する前提でよい
- 第一段階では `links.recording` が YouTube URL なら、それを埋め込みに使う
- `detail.video` は将来 `YouTube以外` や `個別設定` が必要になった時の拡張枠と考える
- `detail.overview` は 2〜4文程度の短い要約を基本にする
- `detail.keyPoints` は 3点前後を上限にする
- 初期入力の必須項目は `タイトル / 日付 / 話者 / テーマ / YouTube URL / 資料URL` を中心に考える

## 8. 表示の考え方
- 動画は `iframe` を最初から埋め込まず、`再生する` を押した時に読み込む
- 埋め込み先は原則 `youtube-nocookie.com` を使う
- PDF はページ内埋め込みではなく、まずは別タブ表示で始める
- 目次がある場合だけ時間リンクを表示する
- 動画ブロックの近くに `再生時に YouTube が読み込まれ、外部サービスへ情報が送信される場合がある` と明示する
- 同じ場所から `プライバシーポリシー` へ移動できるようにする
- 自動再生はしない

これでページの重さを抑えられます。

## 9. 要約の作り方
動画がある回は、可能なら `動画内容ベース` で要約を作ります。

基本フロー:
1. YouTube URL を入れる
2. 字幕があれば取得する
3. 字幕が無い場合は文字起こしを検討する
4. その内容を元に `overview` と `keyPoints` の案を作る
5. 必要なら `chapters` も追加する

要約トーン:
- 現在の `summary` と同じく短く、実務で何を学べるかが分かる書き方にする
- 宣伝調ではなく、教育コンテンツとして落ち着いた文体にする
- 1回あたりの負荷を上げすぎないため、最初は `overview + keyPoints` を優先する
- 字幕が取れない回は、取得できる情報だけで仮要約を作り、公開前後に見直す

## 10. 詳細ページのざっくり構成
第一段階では、必要十分な構成に絞ります。

```text
[パンくず]
[タイトル]
[日付 / テーマ / 話者]

[概要]
[再生エリア]
[資料リンク]
[学習ポイント]
[目次]
[関連アーカイブ]
[注意書き]
[利用規約 / プライバシーポリシー / 問い合わせ]
```

表示ルール:
- `概要` と `再生エリア` を上側に置く
- `資料リンク` は動画のすぐ下に置く
- `学習ポイント` と `目次` はデータがある時だけ表示
- `関連アーカイブ` は同テーマから自動選出でよい
- `利用規約 / プライバシーポリシー / 問い合わせ` はページ下部で常に見えるようにする

## 11. セキュリティ設計の前提
詳細ページは将来 `認証付き資料` や `会員限定コンテンツ` を扱う可能性があるため、以下を先にルール化します。

### 返してよい情報の基準
- ブラウザに返すデータは、閲覧者に見えてよい内容だけに限定する
- メールアドレス、ログインID、内部メモ、限定公開URL、権限判定用の内部情報はレスポンスに含めない
- `id` や `slug` を使っても、そこから個人情報や管理情報を逆引きできない構造にする

### 個人情報と機密情報の責務分離
- 個人情報、認証状態、権限チェック、限定公開判定はサーバーサイドで扱う
- クライアント側は表示のための最小データだけを受け取り、認可ロジックや秘密値を持たない
- 管理画面や会員機能を追加しても、個人情報を `localStorage` や公開 JS に保存しない

### 実装時の安全基準
- XSS 対策: 要約、目次、資料名などの表示はテキストとして扱い、HTML を注入しない
- XSS 対策: 外部由来の埋め込み URL は許可ドメインを絞って検証する
- CSRF 対策: お気に入り登録、コメント、閲覧状態保存などの更新系 API を作る場合は CSRF トークンを必須にする
- SQL インジェクション対策: 詳細取得や検索 API を DB 化する場合は必ずプレースホルダ付きクエリを使う
- 機密情報管理: APIキー、環境変数、限定配信用トークンはハードコードせず、サーバー側だけで扱う

### 動画・資料導線で注意すること
- 限定公開の動画や資料を扱う場合、直リンクをそのまま配布せず、サーバー側で認可後に配信する
- 埋め込みや外部リンクを追加する場合、返却 URL に署名付きトークンや内部パラメータを露出させない
- 将来アクセスログを取る場合も、個人情報は最小限にし、不要なクエリ文字列を保存しない

## 12. 運用の考え方
最初から全部埋める前提にはしません。

初期運用:
- 詳細ページを作る
- 動画 / 資料 / 概要だけ入れる
- ただし公開前に `患者情報 / 著作権 / 登壇者公開可否 / YouTube 埋め込み可否` を確認する

次の段階:
- 学習ポイントを追加
- 目次を追加

この順なら更新負荷が上がりすぎません。

## 13. 実装前に決めること
1. 目次は手入力にするか
2. 埋め込み不可動画の fallback 文言をどうするか
3. 字幕が取れない動画の要約をどこまで手動で補うか

現時点の結論:
- `詳細を見る` ボタンは不要
- タイトルリンクと即時アクションボタンの併用で十分

## 14. 推奨方針
- 第一段階は `archive/index.html?id=...`
- 一覧ページは今のまま軽く保つ
- 動画は `YouTube 埋め込み + 遅延読み込み`
- `detail` は optional
- 要約は `短めの overview + 3点前後の keyPoints`
- 手入力は最小限、足りない情報は可能な範囲で生成補助する
- OGPの個別最適化は第二段階で考える
- 将来認証や API を追加しても、個人情報はサーバー側でのみ扱い、XSS / CSRF / SQL インジェクション対策を実装前提にする
