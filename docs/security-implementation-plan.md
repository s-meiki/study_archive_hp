# セキュリティ実装メモ

## 1. 現在の前提
- 現状は `public/` をそのまま配布する静的サイト
- `admin/` はローカル運用用で、公開サーバー上の認証機能はまだ持たない
- そのため、`public/` に置いたデータは誰でも取得できる前提で扱う

## 2. いま守ること
- `public/data/site-content.js` には公開してよい情報だけを載せる
- メールアドレス、ログインID、権限情報、内部メモ、APIキー、トークン、環境変数の値は置かない
- 外部リンクや入力値はそのまま DOM に差し込まず、安全な URL / テキストとして扱う
- 公開前に `python3 scripts/check_public_security_basics.py` を実行する
- 公開資料の前に `python3 scripts/check_public_uploads_metadata.py` を実行し、Finder 由来メタデータや不要ファイルが残っていないことを確認する
- 静的ページでは `meta http-equiv=\"Content-Security-Policy\"` を入れ、`script-src 'self'` を基本にする
- 将来サーバー配信に移る場合は、CSP は `meta` ではなくレスポンスヘッダーへ移し、必要なら `frame-ancestors` などもヘッダー側で管理する

## 3. 将来サーバー側へ移す責務

### 認証・個人情報
- メールアドレス、ログイン情報、パスワードリセット、セッション管理はサーバー側で扱う
- クライアント側には `表示に必要な最小情報` だけを返す
- 権限判定はフロントで行わず、サーバー側で強制する

### 管理機能
- アーカイブ追加・更新・削除は将来 API 化する場合、認証付き管理 API に分離する
- 更新系 API は CSRF トークン、`SameSite` Cookie、`Origin / Referer` 検証を前提にする
- 変更履歴と監査ログを残す

### 限定公開資料
- 会員限定や院内限定の資料は `public/uploads/` に直接置かない
- サーバー側で認可後に配信する
- 署名付き URL や一時トークンを使う場合も、生成はサーバー側だけで行う

### データベース
- SQL は文字列連結せず、プレースホルダまたは ORM のバインドを使う
- タイトル、概要、URL、検索条件などはサーバー側で形式・長さ・許可値を検証する
- 公開 API は `title / summary / speaker / date / assets / links` など必要最小限の項目に絞る

### 機密情報
- APIキー、DB 接続情報、環境変数、配信用トークンはサーバー環境変数またはシークレット管理で扱う
- リポジトリ、公開 JS、HTML コメント、サンプル値に実値を残さない

## 4. 次にやる順番
1. 公開データと DOM 描画の安全化を継続する
2. 公開前チェックを運用に組み込む
3. 将来ログイン機能を入れる時点で、管理 API と公開 API を分離する
4. 限定公開資料が必要になった時点で、配信をサーバー経由へ切り替える
